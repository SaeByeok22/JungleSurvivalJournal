# Pintos 대비 진단 테스트

> 반영 사항: 네가 말한 것처럼 **KAIST x86-64 기반 Pintos에서 “파일 시스템 구현(Project 4)”은 안 한다**는 전제하에, **파일시스템 구현 문제는 제외**했어.  
> 다만 Project 2에서의 **파일 관련 시스템콜/FD 테이블/동시성 이슈**는 그대로 중요하니 그쪽은 포함했어.

## 안내

- 목적: **Pintos(Threads / User Programs / Virtual Memory)에서 막히는 지점을 수치화**해서, 이후에 “뭘 준비해야 ROI가 큰지”를 정확히 잡기 위함.
- 권장 소요 시간: 60–90분  
- 규칙(권장): **검색/책 없이** 먼저 풀어봐. 모르겠으면 “모르겠음 + 추측”도 적어줘(진단 정확도가 올라감).
- 빠른 버전(30분): 각 섹션의 **(★)** 문제만 먼저.

## 문제

### C / 포인터 / 메모리 안전

**Q1 (★, 6점)** 다음 코드의 출력/동작을 **정확히** 설명해줘. UB(정의되지 않은 동작)가 있으면 “UB”라고 쓰고, 왜 UB인지 적어줘.

```c
#include <stdio.h>

int main(void) {
    int a[3] = {10, 20, 30};
    int *p = a;

    printf("%d\n", *p++);
    printf("%d\n", *++p);
    printf("%d\n", *(p + 0));
    return 0;
}
```

**Q2 (★, 6점)** 다음 코드가 출력하는 값(또는 UB 여부)을 답하고, 이유를 적어줘.

```c
#include <stdio.h>
#include <string.h>

int main(void) {
    char s[4];
    strcpy(s, "ABCD");
    printf("%zu %s\n", strlen(s), s);
    return 0;
}
```

**Q3 (6점)** 다음 구조체의 `sizeof(struct S)`가 64-bit 리눅스(gcc, 일반적인 정렬 규칙)에서 얼마일지 추정하고, **패딩이 어디 들어가는지** 설명해줘.

```c
struct S {
    char  c;
    int   x;
    short y;
    char  d;
};
```

**Q4 (★, 6점)** 아래 함수는 “배열 길이를 구하는 함수”처럼 보이지만 실제로는 문제가 있다.  
문제가 무엇인지, 고치려면 어떤 인터페이스가 되어야 하는지 설명해줘.

```c
#include <stddef.h>

size_t len(int a[]) {
    return sizeof(a) / sizeof(a[0]);
}
```

**Q5 (6점)** 다음 코드는 의도는 “힙에 int 하나를 만들고 7을 저장”이야.  
문제점을 찾고, 수정된 코드를 제시해줘.

```c
#include <stdlib.h>

int main(void) {
    int *p = malloc(sizeof(p));
    *p = 7;
    free(p);
    return 0;
}
```

**Q6 (4점)** “포인터/메모리 관련 버그로 크래시가 나는 전형적 원인 3가지”를 예시와 함께 적어줘.  
(예: NULL 역참조, 해제 후 사용(UAF), 범위 밖 접근(OOB), double free 등)

### 프로세스 / 스레드 / 시스템콜(경계 감각)

**Q9 (★, 6점)** “프로세스”와 “스레드”의 차이를 **주소공간/자원 공유 관점**에서 정리해줘.  
아래 항목이 각각 “프로세스 단위 vs 스레드 단위” 중 어디에 더 가까운지도 표시해줘.
- 가상 주소 공간
- 파일 디스크립터 테이블(일반적인 OS 기준)
- 스택
- 레지스터 집합
- 힙
- 동기화(락/세마포어) 객체의 ‘소유’ 개념(대략)

**Q10 (★, 6점)** 유저 프로그램이 시스템콜을 호출할 때(예: `read`, `write`)  
**유저 모드 → 커널 모드 → 유저 모드**로 돌아오는 “대표 흐름”을 단계로 써줘.  
(정확한 CPU 세부는 몰라도 되지만, *어떤 정보가 어디로 전달/검증*되는지가 핵심)

**Q11 (4점)** 커널에서 유저 포인터(사용자 메모리 주소)를 받았을 때, 왜 “그냥 역참조하면 위험”한지 설명하고, 안전하게 처리하기 위한 원칙을 3가지 적어줘.  
(예: 검증, 예외처리, 접근 방식 등)

### 동기화 / 스케줄링(Phase 1 핵심)

**Q12 (★, 8점)** 아래 코드는 멀티스레드에서 문제가 생긴다.  
1) 어떤 문제가 생길 수 있는지(가능한 결과)  
2) 최소 수정으로 어떻게 고칠지(락을 어디에 둘지)  
3) 락을 너무 크게 잡을 때 생길 수 있는 부작용(성능/데드락/우선순위 등)  
을 적어줘.

```c
int shared = 0;

void inc1000(void) {
    for (int i = 0; i < 1000; i++) {
        shared++;
    }
}
```

**Q13 (6점)** 데드락이 성립하기 위한 대표 “4조건”을 적고, OS 프로젝트에서 현실적으로 쓰는 회피/완화 전략 2가지를 설명해줘.  
(예: 락 순서 규칙 같은 것)

**Q14 (★, 6점)** “우선순위 역전(priority inversion)”이 무엇인지 설명하고, 이를 완화하는 기법(예: donation)의 핵심 아이디어를 짧게 적어줘.  
가능하면 간단한 시나리오(스레드 A/B/C 우선순위 포함)도 같이.

### 가상 메모리 / Malloc 감각(Phase 2~3 연결)

**Q15 (★, 8점)** “페이지 폴트(page fault)”가 났을 때 OS가 하는 일을 가능한 한 구체적으로 단계로 써줘.  
핵심 포인트:
- 폴트의 원인 분류(대표 케이스들)
- 어떤 자료구조/메타데이터를 보고 판단하는지
- 무엇을 적재/매핑/갱신하는지

**Q16 (8점)** 너라면 Malloc Lab 수준에서 아래 요구를 만족하는 `malloc/free`를 어떻게 설계할지 설명해줘.  
(코드 전체가 아니라 **자료구조 + 알고리즘 + 불변식** 중심)

요구:
- 16바이트 정렬
- free block coalescing(인접 free 합치기)
- splitting(큰 free를 쪼개 할당)
- 성능 고려 1개 이상(예: first-fit/next-fit/segregated list 중 택1)

답안에 꼭 포함:
- 블록 메타데이터 형식(헤더/푸터에 무엇을 넣을지)
- free list(implicit/explicit) 선택 및 이유
- `malloc` 흐름(탐색→분할→표시)
- `free` 흐름(표시→병합→리스트 연결)
- 깨지면 바로 터지는 불변식 2개(예: 정렬, 경계 태그 일치 등)

**Q17 (4점)** 아래 상황에서 **외부 단편화(external fragmentation)**가 커지는 이유를 짧게 설명하고, 이를 줄이는 접근 2가지를 적어줘.  
상황: “작은 블록을 많이 할당/해제하며, 서로 떨어진 free 조각이 많이 생기는 워크로드”

## 채점 루브릭(내가 채점할 기준)

총점 100점.

- C/포인터/메모리 안전: 34점 (Q1~Q6)
  - **정답 + 이유/근거가 정확**하면 만점
  - “대충 감”만 있으면 부분점수(절반 이하)
  - UB를 UB로 못 잡으면 큰 감점
- 빌드/디버깅: 10점 (Q7~Q8)
  - 옵션 의미를 단순 나열이 아니라 “왜 쓰는지(디버그/성능/경고 정책)”까지 말하면 고득점
- 프로세스/스레드/시스템콜: 16점 (Q9~Q11)
  - “경계(유저/커널, 주소공간, 검증)”를 정확히 이해하면 고득점
- 동기화/스케줄링: 20점 (Q12~Q14)
  - race를 결과로 설명 + 정확한 락 위치 + 우선순위 역전 개념까지 연결하면 고득점
- VM/Malloc: 20점 (Q15~Q17)
  - “원인→판단→조치(매핑/적재/갱신)” 흐름이 있으면 고득점
  - malloc은 자료구조/불변식이 없으면 감점

## 점수 해석 가이드(다음 단계에서 ROI를 결정할 구간)

- **80–100점:** 바로 Pintos 들어가도 됨. (부족한 영역만 핀포인트 보강)
- **65–79점:** “돌리면서 배우는” 건 가능하지만, 특정 영역이 발목 잡을 확률 큼 → **약점 영역 집중 보강**이 ROI 최고
- **50–64점:** Pintos 시작은 가능하나, 디버깅/동기화/유저포인터 검증에서 시간을 크게 잡아먹을 가능성 높음
- **0–49점:** 지금 들어가면 프로젝트가 “OS가 아니라 C 디버깅”이 될 확률 큼 → **C/메모리/디버깅을 먼저** 잡는 게 ROI 최댓값

영역별 위험 신호(대략):
- C 영역이 **24/34 미만**: 포인터/UB/메모리 안정성부터 보강하지 않으면 Pintos가 매우 비싸짐
- 동기화가 **12/20 미만**: Phase 1에서 donation/락 설계에서 크게 흔들릴 확률 높음
- 시스템콜/유저포인터가 **10/16 미만**: Phase 2에서 “왜 자꾸 죽지?”가 반복될 확률 높음
- VM/Malloc이 **12/20 미만**: Phase 3에서 page fault/mmap/swap 개념이 계속 헷갈릴 확률 높음
